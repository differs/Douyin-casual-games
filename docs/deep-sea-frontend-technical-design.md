# 深海吞噬进化前端技术设计文档

## 文档目标

定义首版前端实现方式、工程结构、关键模块和开发边界。

目标只有一个：

**以最快速度做出可玩、可接广告、可接后端的抖音小游戏前端。**

## 技术选型

- 引擎：`Cocos Creator`
- 语言：`TypeScript`
- 平台：抖音小游戏

## 选型结论

前端不做自定义引擎，不做原生裸写。

直接使用 `Cocos Creator` 的原因：

- 2D 游戏开发效率高
- 场景和节点编辑快
- 碰撞、动画、资源管理成熟
- TypeScript 工程化成本低
- 后续导出抖音小游戏路径明确

## 首版前端职责

前端负责：

- 页面与 UI
- 输入控制
- 主循环
- 地图内实体生成
- 吞噬逻辑
- 碰撞检测
- 成长升级
- 分数计算
- 广告触发
- 后端 API 调用

前端不负责：

- 排行榜复杂排序
- 权威对局校验
- 重风控

## 页面结构

首版只保留 4 个页面或弹层：

1. 首页
2. 游戏主场景
3. 结算页
4. 复活弹层

## 页面说明

### 首页

内容：

- 开始游戏按钮
- 最高分展示
- 音效开关
- 皮肤入口占位

### 游戏主场景

内容：

- 海洋背景
- 玩家鱼
- AI 鱼与食物
- 当前分数
- 当前成长阶段
- 暂停入口

### 结算页

内容：

- 本局分数
- 最高分
- 达成阶段
- 再来一局
- 双倍奖励广告按钮

### 复活弹层

内容：

- 是否复活
- 激励广告按钮
- 放弃按钮

## 场景与目录建议

建议工程结构如下：

```text
assets/
  scenes/
  scripts/
    core/
    config/
    game/
    ui/
    services/
    model/
  resources/
    prefab/
    sprite/
    audio/
    effect/
```

## 目录职责

### `scripts/core`

- 游戏状态机
- 全局事件总线
- 常量定义
- 工具函数

### `scripts/config`

- 成长阶段配置
- 刷新配置
- 分数配置
- 广告奖励配置

### `scripts/game`

- 玩家控制器
- AI 鱼控制器
- 刷新管理器
- 碰撞管理器
- 成长系统
- 分数系统

### `scripts/ui`

- 首页控制器
- HUD 控制器
- 结算控制器
- 复活弹层控制器

### `scripts/services`

- 抖音平台封装
- 广告服务
- HTTP 请求服务
- 本地存储服务

### `scripts/model`

- 用户数据模型
- 对局结果模型
- 存档模型

## 核心系统设计

## 1. 游戏状态机

建议状态：

- `Boot`
- `Home`
- `Playing`
- `RevivePending`
- `GameOver`

作用：

- 简化场景切换
- 防止 UI 与游戏状态混乱

## 2. 玩家控制

实现原则：

- 手指拖动控制方向
- 玩家鱼自动前进
- 松手后保持缓动惯性

首版不做：

- 复杂冲刺
- 技能按钮
- 多输入模式

## 3. 实体类型

首版只需要三类实体：

- 玩家鱼
- AI 鱼
- 静态食物

### 玩家鱼

字段建议：

- `stage`
- `speed`
- `radius`
- `score`
- `alive`

### AI 鱼

字段建议：

- `stage`
- `speed`
- `radius`
- `move_pattern`
- `score_value`

### 静态食物

字段建议：

- `type`
- `radius`
- `score_value`

## 4. 刷新系统

首版使用简单权重刷新即可。

需要三个刷新池：

- 食物池
- 小中型鱼池
- 大型危险鱼池

控制维度：

- 场上最大数量
- 每阶段权重
- 随时间增加危险鱼比例

首版不要做复杂生态逻辑。

## 5. 吞噬与碰撞

### 基础规则

- 玩家碰到低级目标则吞噬成功
- 玩家碰到同级或高级鱼则失败
- AI 鱼之间不做相互吞噬

### 判定方式

首版直接采用圆形碰撞半径判定。

原因：

- 实现简单
- 性能可控
- 足够匹配卡通鱼类素材

## 6. 成长系统

成长由经验阈值控制。

建议配置项：

- `stage_id`
- `name`
- `exp_required`
- `player_scale`
- `speed`
- `eat_radius`

升级触发后：

- 替换外观
- 调整半径
- 播放升级特效
- HUD 刷新阶段显示

## 7. 分数系统

首版分数公式尽量简单：

- 吞食物得分
- 吞鱼得分
- 存活时间得分

建议公式：

```text
score = food_score + fish_score + survival_seconds * time_factor
```

不要在首版加入过多乘区。

## 8. 相机与地图

建议使用单场景大地图加跟随相机。

处理方式：

- 玩家居中
- 相机平滑跟随
- 地图边界限制

视觉上通过背景渐变和装饰分层表达深海感。

首版不做真正多海域切换。

## 9. UI 系统

HUD 最少显示：

- 当前分数
- 当前阶段
- 成长进度条

可选显示：

- 最高分

首版 UI 原则：

- 大按钮
- 少文本
- 强反馈

## 10. 广告系统

广告系统封装成独立服务：

- `showReviveAd()`
- `showDoubleRewardAd()`

前端只处理：

- 按钮点击
- 广告回调
- 成功后调用后端奖励接口

首版不要把广告逻辑散在 UI 代码里。

## 11. 后端通信

前端只对接这几类接口：

- 登录
- 拉取资料
- 拉取存档
- 保存设置
- 提交分数
- 领取广告奖励
- 上报埋点

建议统一封装 `ApiService`：

- 自动带 token
- 统一错误处理
- 统一重试策略

## 12. 本地缓存

建议本地缓存内容：

- token
- 音效设置
- 最近一次最高分缓存

不要把奖励结算完全依赖本地，避免状态错乱。

## 数据配置建议

建议配置文件拆分为：

- `stage-config`
- `spawn-config`
- `score-config`
- `ad-config`

首版先用本地静态配置文件，不做服务端动态下发。

## 性能与包体原则

首版必须坚持：

- 资源复用
- 贴图数量少
- 音效数量少
- 简单粒子，不堆特效

优化重点：

- 对象池复用鱼和食物
- 限制场上最大实体数
- 避免频繁创建销毁节点

## 最小素材方案

主角：

- 8 个阶段图

AI：

- 6 到 8 个鱼图

食物：

- 3 到 4 个小素材

装饰：

- 气泡、珊瑚、石头、海草

特效：

- 吞噬
- 升级
- 失败

## 开发顺序建议

### 第一步

先做一个无 UI 原型：

- 玩家移动
- 食物吞噬
- AI 鱼碰撞
- 升级

### 第二步

补完整局流程：

- 首页
- HUD
- 失败
- 结算

### 第三步

接平台与后端：

- 登录
- 提交分数
- 广告

### 第四步

做调优：

- 刷新节奏
- 难度曲线
- 升级手感

## 首版不做事项

- 多模式
- 联网同步
- 排行榜页
- 复杂鱼类 AI
- 多技能系统
- 大量皮肤

## 最终前端结论

首版前端目标不是“架构多漂亮”，而是：

- 尽快有原型
- 尽快可玩
- 尽快可接广告
- 尽快可提审

因此技术路线保持简单直接：

**Cocos Creator + TypeScript + 本地对局逻辑 + 少量后端接口。**
