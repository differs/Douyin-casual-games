# 深海吞噬进化开发任务拆解文档

## 文档目标

把当前方案拆成可以直接执行的开发任务，避免“知道要做什么，但不知道先做哪一块”。

拆解原则：

- 按交付顺序，而不是按想法顺序
- 优先做能形成闭环的模块
- 前后端并行，但以前端原型优先

## 总体阶段

建议分成 5 个阶段：

1. 项目初始化
2. 前端核心玩法原型
3. 前端完整闭环
4. 后端与平台接入
5. 联调、调优、提审准备

## 阶段一：项目初始化

目标：

- 建立前端工程
- 建立后端工程
- 确定基础目录与规范

### 前端任务

- 创建 `Cocos Creator` 工程
- 确定抖音小游戏导出配置
- 建立基础目录：
  - `scenes`
  - `scripts/core`
  - `scripts/config`
  - `scripts/game`
  - `scripts/ui`
  - `scripts/services`
  - `resources`
- 建立全局常量文件
- 建立基础事件总线
- 建立本地配置文件结构

### 后端任务

- 创建 Rust 工程
- 建立 `Axum` 服务基础骨架
- 接入配置读取
- 接入日志
- 接入 PostgreSQL 连接
- 建立路由结构
- 建立统一响应结构

### 验收结果

- 前后端工程都能启动
- 前端可导出到小游戏目标
- 后端可启动一个健康检查接口

## 阶段二：前端核心玩法原型

目标：

- 在不依赖后端的前提下，先做出可玩原型

这是最优先阶段，没有这个阶段，后续都不该展开。

### 任务 2.1 玩家移动

- 实现手指拖动控制方向
- 实现玩家自动前进
- 实现松手后缓动

验收：

- 玩家能平滑移动
- 不出现明显抖动和失控

### 任务 2.2 食物与 AI 鱼实体

- 定义食物实体
- 定义 AI 鱼实体
- 实现基础 prefab
- 实现基础展示与刷新

验收：

- 场景中能持续生成食物和鱼

### 任务 2.3 碰撞与吞噬

- 实现圆形碰撞检测
- 实现吞噬判定
- 实现危险鱼碰撞失败

验收：

- 吞噬规则正确
- 碰撞失败稳定

### 任务 2.4 成长升级

- 实现成长经验条
- 实现阶段阈值配置
- 实现升级时角色外观和半径变化

验收：

- 玩家能至少升到多个阶段
- 升级后吞噬范围有变化

### 任务 2.5 分数系统

- 统计吞食物得分
- 统计吞鱼得分
- 统计生存时间得分

验收：

- 局内分数实时增长

### 验收结果

- 无后端下可完成一整局
- 有基础成长反馈
- 失败后可进入本地结算

## 阶段三：前端完整闭环

目标：

- 补齐 UI、结算、广告占位、本地存档

### 任务 3.1 首页

- 制作首页 UI
- 展示开始按钮
- 展示最高分占位
- 展示音效开关

### 任务 3.2 HUD

- 当前分数
- 成长条
- 当前阶段

### 任务 3.3 结算页

- 展示本局分数
- 展示最高分
- 展示再来一局按钮
- 展示双倍奖励入口

### 任务 3.4 复活弹层

- 失败时弹出
- 可选复活或放弃
- 每局限制一次

### 任务 3.5 本地存档

- 本地缓存最高分
- 本地缓存音效开关

### 任务 3.6 广告假流程

- 用 mock 方式先模拟广告成功回调
- 跑通复活和双倍奖励流程

### 验收结果

- 首页 -> 游戏 -> 失败 -> 复活/结算 -> 再来一局 全链路打通

## 阶段四：后端与平台接入

目标：

- 打通抖音登录
- 打通服务端接口
- 替换本地假数据

### 后端任务 4.1 用户与登录

- 实现 `POST /api/login`
- 完成本地用户创建或查询
- 签发 token

### 后端任务 4.2 用户资料与存档

- 实现 `GET /api/user/profile`
- 实现 `GET /api/user/archive`
- 实现 `POST /api/user/archive`

### 后端任务 4.3 分数提交

- 实现 `POST /api/score/submit`
- 更新最高分
- 返回奖励结果

### 后端任务 4.4 广告奖励

- 实现 `POST /api/ad/reward`
- 实现去重逻辑
- 实现每日限制逻辑

### 后端任务 4.5 埋点

- 实现 `POST /api/event/report`

### 前端任务 4.6 平台服务封装

- 封装登录能力
- 封装广告能力
- 封装请求 token 注入

### 前端任务 4.7 联调替换

- 替换本地最高分为服务端数据
- 替换本地广告 mock 为真实奖励确认
- 替换本地结算为服务端提交

### 验收结果

- 真机下能完成登录
- 分数能提交
- 最高分可回读
- 广告奖励可发放

## 阶段五：联调、调优、提审准备

目标：

- 让首版达到可测试、可提审状态

### 任务 5.1 首局体验优化

- 让玩家前 10 秒必能吞到东西
- 让玩家前 20 秒大概率完成首次升级

### 任务 5.2 数值调平衡

- 调整刷新权重
- 调整成长阈值
- 调整大鱼出现节奏

### 任务 5.3 埋点验证

- 验证关键事件都能上报
- 验证字段正确

### 任务 5.4 稳定性检查

- 连玩 10 局
- 广告失败回退测试
- 弱网下结算测试

### 任务 5.5 提审材料准备

- 游戏图标
- 游戏名称
- 启动图
- 文案统一

### 验收结果

- 首版稳定
- 功能闭环完整
- 具备提审条件

## 并行分工建议

如果是 2 到 3 人小团队，可按以下分工：

### 前端

- 负责全部玩法和 UI
- 负责平台 API 封装
- 负责广告接入

### 后端

- 负责登录、存档、分数、奖励、埋点
- 负责数据库与部署

### 兼任/产品

- 负责配置表
- 负责数值调优
- 负责素材整理
- 负责提审资料

## 开发优先级排序

严格按下面顺序推进：

1. 玩家移动
2. 吞噬与失败
3. 成长升级
4. 分数与结算
5. 首页与 HUD
6. 复活与双倍奖励流程
7. 后端登录与存档
8. 分数与广告奖励接口
9. 埋点与调优

## 风险提醒

最容易浪费时间的点有 4 个：

1. 过早打磨美术而不是先做原型
2. 过早设计复杂后端
3. 过早引入皮肤、商店、任务系统
4. 没先把首局体验调顺

## 当前建议

现在最正确的执行方式不是“同时做所有模块”，而是：

先把 **前端单局原型** 在最短时间内做出来。

只有这一步成立，后续后端与平台接入才有价值。
