# 深海吞噬进化项目方案

## 项目目标

开发一个可快速上线验证的抖音小游戏：

- 题材：深海吞噬进化
- 玩法：单局无限吞噬
- 变现：纯广告
- 后端：Rust
- 前端：以“最快做出来”为第一原则

本方案重点不是做大而全，而是做一个 **能尽快跑通、可接广告、能验证留存与复玩** 的首版。

## 核心结论

### 后端

后端使用 **Rust**，但首版必须严格控制职责，只承担：

- 登录态换取与用户标识映射
- 基础用户存档
- 最高分记录
- 每日广告奖励发放校验
- 基础埋点接收

首版后端 **不参与实时对局**，不做帧同步，不做房间，不做战斗判定，不做复杂排行榜。

### 前端

前端建议直接采用 **Cocos Creator** 开发抖音小游戏。

原因：

- 对 2D 吞噬类玩法足够
- 场景、碰撞、动画、UI、资源管理现成
- 导出小游戏比自写原生渲染更快
- 未来做特效、广告节点、结算页也更顺手

结论很明确：

- 如果目标是“怎么快怎么做”，前端不要自建渲染框架
- 直接上 **Cocos Creator + TypeScript**

## 首版产品范围

### 保留内容

## 技术架构

## 整体结构

分为两部分：

1. 前端小游戏客户端
2. Rust 轻后端服务

客户端负责：

- 游戏主循环
- 输入控制
- 碰撞判定
- 吞噬与成长计算
- UI 与广告触发
- 本地缓存

后端负责：

- 用户身份与存档
- 分数上报
- 广告奖励校验
- 数据统计接口

## 为什么不把对局放后端

因为这是抖音极简小游戏首版，目标是低成本验证，不是强对抗竞技。

把对局完全放前端有几个实际好处：

- 开发快
- 调试快
- 服务器成本低
- 架构简单
- 能尽快提审

首版允许一定程度的客户端可控风险，后续如果数据起来，再逐步把高价值结算逻辑收回服务端。

## 前端方案

## 技术选型

- 引擎：`Cocos Creator`
- 语言：`TypeScript`
- 平台目标：抖音小游戏

## 前端模块划分

建议按以下结构组织：

- `scenes/`
- `scripts/core/`
- `scripts/game/`
- `scripts/ui/`
- `scripts/services/`
- `resources/`

建议职责如下：

- `core`
  - 游戏状态机
  - 常量配置
  - 事件系统
- `game`
  - 玩家控制
  - 鱼类生成
  - 碰撞检测
  - 成长升级
  - 得分计算
- `ui`
  - 首页
  - HUD
  - 结算页
  - 复活弹层
- `services`
  - 抖音平台 API 封装
  - 广告封装
  - 后端请求封装

## 前端首版关键设计

### 1. 对局逻辑本地化

首版所有游戏内计算都在客户端完成：

- 移动
- 刷怪
- 吞噬
- 升级
- 分数

### 2. 数据驱动配置

把成长阶段、鱼类权重、刷新频率写成配置表。

这样做的好处：

- 调平衡不改核心逻辑
- 后续更容易扩展

建议最少配置：

- 主角成长阶段表
- AI 鱼权重表
- 分数规则表
- 广告奖励表

### 3. 资源策略

首版严格控资源：

- 主角阶段 8 个
- AI 鱼 6 到 8 个
- 背景 1 套
- 装饰少量复用

尽量控制包体，避免前期就因为资源膨胀拖慢启动。

## 后端方案

## 技术栈建议

- 语言：`Rust`
- Web 框架：`Axum`
- 数据库：`PostgreSQL`
- 缓存：首版可不接 Redis
- 部署：单机容器即可

为什么这样选：

- `Axum` 轻量直接，适合 API 服务
- `PostgreSQL` 足够稳，后续扩展也方便
- 首版并发规模不大，不必一开始引入太多组件

## 后端模块

建议拆成以下几块：

- `auth`
- `user`
- `archive`
- `score`
- `ad_reward`
- `analytics`

### auth

职责：

- 通过抖音小游戏登录态换取平台用户信息
- 建立本地用户 ID

### user

职责：

- 返回用户基础资料
- 返回当前金币、最高分、已解锁皮肤

### archive

职责：

- 保存基础游戏存档
- 保存设置项，例如音效开关

### score

职责：

- 接收单局结算分数
- 更新个人最高分

### ad_reward

职责：

- 接收激励广告回调或奖励确认
- 防止同一奖励被重复领取

### analytics

职责：

- 接收客户端埋点
- 用于后续看漏斗与留存

## 后端接口范围

首版只建议做以下接口：

- `POST /api/login`
- `GET /api/user/profile`
- `GET /api/user/archive`
- `POST /api/user/archive`
- `POST /api/score/submit`
- `POST /api/ad/reward`
- `POST /api/event/report`

这个接口集已经够首版使用。

不要一开始就做：

- 复杂排行榜分页
- 多维度榜单
- 邮件系统
- 活动系统
- GM 系统

## 客户端与服务端边界

首版建议按这个边界执行：

客户端可信但不过度信任。

### 客户端直接决定

- 玩家移动
- 对局得分
- 吞噬过程
- 升级过程

### 服务端负责落库与简单校验

- 最高分更新
- 奖励发放去重
- 用户基础资产保存

### 服务端做最小风控

- 分数上限粗校验
- 广告奖励频次校验
- 同日奖励去重

不要试图首版就做重反作弊，否则交付速度会明显下降。

## 开发阶段建议

## 第一阶段：纯前端可玩原型

目标：

- 不接后端
- 不接真实广告
- 先跑通核心玩法

完成内容：

- 移动
- 吞噬
- 升级
- 刷新
- 结算

这是最关键的一步，先确认“玩起来顺不顺”。

## 第二阶段：接入平台与后端

目标：

- 打通登录
- 打通存档
- 打通分数提交
- 接入广告位

完成内容：

- 平台登录
- API 封装
- 分数提交
- 广告奖励结算

## 第三阶段：调优与提审

目标：

- 做基础新手引导
- 调平衡
- 做埋点
- 处理审核问题

完成内容：

- 首局体验优化
- 结算节奏优化
- 崩溃和弱机排查
- 文案与图标整理

## MVP 页面清单

首版只做 4 个页面/场景：

1. 首页
2. 游戏场景
3. 结算页
4. 复活弹层

这已经足够。

## 广告设计

首版只放 2 个核心广告点：

### 1. 失败复活

- 每局最多 1 次
- 观看激励广告后原地复活

### 2. 结算双倍奖励

- 看广告双倍金币

这两个点最自然，优先保留。

不开更多广告位，避免首版体验过重。

## 埋点建议

首版最少埋以下事件：

- `game_start`
- `game_over`
- `revive_click`
- `revive_success`
- `double_reward_click`
- `double_reward_success`
- `stage_upgrade`
- `session_duration`

有这些数据，就能先看核心漏斗。

## 排期建议

如果目标是“尽快出首版”，建议按下面节奏推进：

### 第 1 周

- 确定素材风格
- 搭 Cocos 工程
- 完成移动、吞噬、成长

### 第 2 周

- 完成 UI
- 完成结算
- 完成广告占位和假数据流程

### 第 3 周

- Rust 后端接口
- 接入登录与存档
- 接入分数上报

### 第 4 周

- 调平衡
- 补埋点
- 提审准备

## 当前技术决策

最终建议明确如下：

- 游戏题材：`深海吞噬进化`
- 前端实现：`Cocos Creator + TypeScript`
- 后端实现：`Rust + Axum + PostgreSQL`
- 对局逻辑：`前端本地执行`
- 变现方式：`纯激励广告`
- 首版目标：`4 周内做出可提审 MVP`
